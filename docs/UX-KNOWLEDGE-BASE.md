# UX 開發知識庫 — 反覆踩過的使用者體驗坑（精華版）

> 本文件由五個 repo（AI-Thermal、5G-RRU Volume、5G-RRU Wind、Thermal-Test-Report-Builder、
> iPAS Exam Simulator）約 300 個 commit 的修 bug 歷史提煉而成。
> **這些是 Tedus 反覆要求過的慣例：開發任何 UI 時預設遵守，不需要他每次重新描述。**
> 每條規則後面括號標註實際踩坑出處，方便回頭查參考實作。

---

## 1. 表單與輸入

1. **輸入中絕不整區重繪。** 使用者打字時若觸發整個表格/區塊 re-render，游標會跳開、
   輸入中斷。事件處理要只更新受影響的 DOM 節點，或在 blur/debounce 後才重繪。
   （Report Builder PR #54：比對頁 Sim Tc 輸入時游標跳開）
2. **重繪或存檔後要還原捲動位置。** 任何 re-render 不可以把使用者彈回頁面頂端。
   （Report Builder PR #39/#40：scroll-to-top 修正）
3. **文字欄位必須支援貼上，表格要支援從 Excel 直接貼多列。** 工程使用情境幾乎都是
   從 Excel 複製資料過來，不能只支援逐格打字。
   （Report Builder PR #37/#39：paste 進文字欄位、Excel paste）
4. **提供 undo/redo。** 編輯器類頁面一開始就要規劃復原機制，不要等做壞了才補。
   （Report Builder PR #37：全頁型別加 undo/redo）
5. **自動帶入的欄位一律「鎖定＋逃生口」。** 由規則推導出來的欄位（例如 TIM Size＝
   凸台大小、IC 距離＝0.5）要鎖成不可誤改，但旁邊給一個 ✂ 解鎖鈕讓進階使用者手動
   覆寫；來源值變動時，鎖定中的欄位要同步跟隨。
   （AI-Thermal PR #163~#165：Pad/Putty/Grease 自動帶入並鎖定）
6. **純參照值顯示為白底黑字純文字，不要用反灰 disabled input。** 反灰輸入框看起來
   「壞掉了」而且難閱讀；不可編輯就不要長得像輸入框。
   （AI-Thermal PR #158：參照鎖定瓦數改純文字顯示）
7. **能自動判定就不要讓使用者手選。** 例如 Verdict 由 Margin 自動推導
   （≥10%→Pass、0–10%→Warning、<0%→Fail），Overall 由各元件 Verdict 彙整；
   空清單有明確語意時直接給結論（空 Compliance 清單＝全數符合＝PASS），
   不要顯示「尚未判定」讓使用者困惑。
   （Report Builder PR #65/#66）
8. **自動帶入邏輯要有 fallback，不能只認一個 happy path。** 例如帶 Meas Tc 時
   Ta=55 沒值就往下找「最高且有填值」的 Ta，只要有量測值就帶得進來。
   （Report Builder PR #64）
9. **鍵盤要能走完流程。** 確認對話框支援 Enter 直接送出。
   （AI-Thermal PR #161：刪除專案密碼框 Enter 即刪）

## 2. 資料一致性與連動更新

1. **同一份資料只有一個事實來源。** 在多分頁工具裡，某欄位若鏡像另一頁的資料
   （如 TH/ME 清單的元件名稱鏡像 Tab1），鏡像端做成唯讀，別讓兩邊可各自改。
   （AI-Thermal PR #149）
2. **改名要連動所有參照。** 元件改名時，標註圖、清單、報告裡引用它的 ref 都要
   一起更新，否則出現孤兒參照。（AI-Thermal PR #151：componentRef 連動更新）
3. **切換下拉/選項時，相依的 UI 要即時更新。** 例如切換元件類別，代表色色塊與
   自動編號要立刻刷新，不能等下一次整頁重繪。（Report Builder PR #68）
4. **重複輸入用「一鍵帶入」消滅。** 跨頁資料（標註頁元件 → 數據頁、跨專案元件
   快選、複製專案、複製標註標籤）都做成可勾選的批次帶入，並把該帶的欄位帶齊，
   不要帶半套讓使用者補漏。
   （Report Builder PR #71；AI-Thermal PR #153~#156「快選」、PR #155 帶齊 Tab1 欄位）
5. **建議值採 click-to-apply。** 自動建議（如由 OSM 建物密度推 α、遮蔽係數）
   顯示成卡片＋一鍵套用，不要直接改使用者的輸入。（Wind Tool：auto-suggest 卡片）

## 3. 螢幕／預覽／PDF 三方一致（報告類工具最大坑區）

1. **改編輯器就要同步改 PDF builder，同一個 commit 內完成。** 樣式（深藍表頭、
   框線、字級）、資料欄位、色塊，三方（編輯器/預覽/PDF）任何一方單獨改動都是 bug。
   （Report Builder PR #46~#55 整串：表頭深藍背景遺失、Sensor 欄舊值、樣式同步）
2. **預覽/PDF 讀 state 不讀 DOM 快照，防 stale value。** 欄位更新後預覽仍顯示
   舊值，多半是 PDF builder 讀了未同步的副本。（Report Builder PR #53）
3. **共用的視覺常數抽成一份。** 類別色、字級這種三方都用的值，抽
   `CATEGORY_OPTIONS` / `catColorOf()` 之類的共用函式，不要在編輯器、預覽、PDF
   各寫一份三元運算。（Report Builder PR #67）
4. **PDF 分頁上限用「實測座標」校準，不要相信理論量測。** CJK webfont 的字寬
   量測會低估列高，導致最後一列壓到頁尾；要以實際瀏覽器量到的列底座標校準
   換頁門檻，且 CJK 字型要以實際用到的字元預載子集。
   （Report Builder PR #60~#63：換頁上限 532、CJK 子集預載）
5. **圖片變換（旋轉/翻轉/位移）編輯器與 PDF 用同一個 transform 字串**，
   輸出才會一致。（Report Builder：標註頁圖片工具）
6. **PDF 裡的圖片要保持長寬比**，不可拉伸。（Report Builder：logo 拉伸修正）

## 4. 版面配置與小螢幕（使用者是 17 吋筆電）

1. **小螢幕策略是「加寬整頁」，不是硬塞。** 欄位多的表格頁在小螢幕上掙脫 A4
   比例、回收左右留白（scrollable-edit 模式），讓所有欄位清楚可讀；PDF 版面不動。
   （Report Builder PR #59、比對頁/結論頁/標註頁陸續沿用同一招）
2. **內容欄（如元件名稱）要給足寬度**，寧可犧牲裝飾性欄位。（Report Builder PR #57）
3. **Header/工具列的元素用流動排列，不要絕對定位。** 絕對定位的徽章會蓋到
   登入 email、擠壓換行；放不下就移到次要位置（如分頁列最右側）。
   （AI-Thermal PR #143/#144：版本徽章重疊與換行）
4. **新增按鈕前先看那一排放不放得下；移動按鈕後要刪乾淨舊的。** 按鈕互相重疊、
   squash-merge 殘留重複按鈕都發生過。（5G-RRU PR #70~#72）
5. **浮動視窗（popover/密碼框）要防跑出畫面外**，定位計算要 clamp 在 viewport 內。
   （AI-Thermal PR #130）
6. **側邊欄與主畫面不要放重複功能的按鈕。**（AI-Thermal PR #136）

## 5. 顏色與視覺

1. **紅色保留給 Fail/錯誤，任何分類色都要避開紅。**（Report Builder PR #67 四色規劃）
2. **分類代表色彼此要明顯區隔**（藍/綠/橘/紫），相近色（紫 vs 藍）要調開；
   螢幕與 PDF 用同一組色值。（Report Builder PR #67）
3. **圖表上的疊加元素（箭頭、標記）要對「所有可能的底色」都有對比。**
   風玫瑰圖箭頭從白→亮綠 #00FF88 就是因為白色在淺色速度帶上看不見。
   （Wind Tool：箭頭改色兩次才修好——選色時把每個底色都列出來檢查）
4. **深色主題做的元件，要回頭檢查淺色主題（反之亦然）。**
   （Wind Tool：badge popover 在 light theme 顏色錯誤）
5. **面板/卡片底色要跟所在區域有對比**，疊在圖片或深色畫布上的面板給白底。
   （Report Builder PR #44：TIM panel 白底）

## 6. 圖片標註／畫布類功能

1. **標註座標一律存「相對圖框」座標（0~1），不要存絕對 px。** 絕對座標在
   重開、換螢幕、縮放時會飄移；點與標籤要用同一套座標系。
   （AI-Thermal PR #145：標籤飄移修正）
2. **圖框要跨螢幕等比縮放貼合**，避免在較小螢幕溢出。（AI-Thermal PR #146）
3. **模式互斥要處理：** 進入「移動/縮放圖片」模式時暫停「點擊新增標註」，
   拖曳不可誤觸新增；控點與按鈕的點擊也不可觸發標註。（Report Builder 標註頁 2/2）
4. **存檔時對帳清理孤兒附件**，確保「畫面顯示的＝資料庫存的」且數量一致。
   （AI-Thermal PR #147：孤兒圖片清理）
5. **標註物件要給：可見的刪除鈕、click-to-edit 標籤、複製鈕（hover 顯示）。**
   （Report Builder：小紅點＋click-to-edit＋複製標籤）
6. 縮放控點放**四角、維持長寬比、固定對角**，不要只給右下角一顆。（Report Builder）

## 7. 鎖定、唯讀與多人協作

1. **唯讀/鎖定狀態要「全面」：** 所有寫入入口（新增專案、編輯鈕、刪除鈕）都要
   反灰停用，漏一顆就是 bug；同時提供「眼睛」唯讀檢視當替代出路。
   （AI-Thermal PR #137/#139）
2. **切換到鎖定狀態前，攔截未儲存變更**，給「儲存/放棄/取消」三選項，
   且所有分頁行為一致。（AI-Thermal PR #138）
3. **顯示「誰」正持有編輯鎖**（置頂橫幅），不要只顯示鎖住了。（5G-RRU PR #63）
4. **鎖的生命週期要誠實：** 閒置逾時主動釋放並退回鎖定 UI；每次寫入前驗證鎖
   仍屬於自己；快過期就續約；被接手時 UI 立刻退回，不能讓使用者以為還持鎖。
   （5G-RRU PR #67 / AI-Thermal PR #170）
5. **beforeunload 只提醒不釋放鎖**（按「留下」後鎖不能飛掉），實際釋放放 pagehide。
   （同上）

## 8. 儲存與破壞性操作

1. **儲存行為要符合心智模型：按任一儲存鍵就是存「全部」**（所有分頁一起存），
   使用者不會記得哪頁還沒存。（AI-Thermal PR #128）
2. **寫後立即重讀不能讓資料「看似消失」。** 存檔後的重載要以剛寫入的內容為準，
   不要被快取/版本比對誤判成衝突而嚇到使用者。（AI-Thermal PR #157）
3. **破壞性操作三件套：** 確認關卡（密碼或對話框）＋連帶清理關聯資料（標註圖、
   參照）＋刻意刪除與異常歸零要能區分（保險絲例外放行）。
   （AI-Thermal PR #161；共用 DB 規則第 5 條）
4. **刪除入口要可見**，不要藏在右鍵或 hover 才出現的地方（表格列給明確刪除鈕）。
   （Report Builder PR #56）
5. **共用資料庫的寫入規則見各 repo CLAUDE.md**（updateDoc 不用 setDoc、
   nested object 先讀再合併、壞檔唯讀保護、ETag 樂觀並發）——那是另一套
   必守規則，這裡不重複。

## 9. 版本更新與快取

1. **使用者會停在舊版網頁不重整。** 部署型工具要有：CI 自動戳版本號（佔位符
   `__APP_VERSION__`＋sed）、前端定時比對 version.json、有新版跳醒目橫幅、
   是最新版時顯示「✓ 最新」徽章徹底靜默。**版本號永遠不手改。**
   （AI-Thermal PR #140~#142；細節見 CLAUDE.md「軟體版本戳記」節）
2. **本地 JS 一律掛 `?v=__APP_VERSION__` 快取戳記**，否則使用者拿到新 HTML
   舊 JS。

## 10. 外部資源與網路

1. **所有外部 fetch 一律包 timeout（AbortController）＋自動重試**（網路錯誤/429/
   5xx 重試一次），寫成 `fetchSafe()` 之類的共用 helper。（Wind Tool H-02）
2. **第三方服務（geocoder、QR code）要處理長輸入、特殊字元的失敗案例。**
   （Wind Tool：長地名 QR code 產生失敗、geocoder retry）
3. **減少使用者打字：** 能從已有資料推建議就推（OSM 建物密度→α/遮蔽係數）、
   能解析貼上的 URL 就解析（Google Maps URL→座標）、加搜尋歷史。（Wind Tool）

## 11. 數字可信度與文案

1. **UI 上每個統計數字要能回答「哪來的」：** 圖表標題下標註取樣範圍
   （如「Sample: hot season daytime 10:00–18:00」）、資料來源（ERA5、OSM）、
   計算公式放 popover。（Wind Tool：sampling scope note、alignment 公式 popover）
2. **文案必須與程式實際行為一致。** 說明寫「只取白天資料」但程式沒過濾，
   就是 bug——改演算法時要同步搜尋 UI 文案與 methodology 文件。（Wind Tool M-02）
3. **長說明做成可折疊**，預設收合，不擠壓主內容。（Wind Tool：footnotes collapsible）
4. **單位要對：** 溫差對使用者顯示 °C 不用 K；改名欄位時螢幕與 PDF 一起改。
   （Wind Tool：K→°C）
5. **雙語（中/英）文案要同步改**，包含 tooltip、methodology、PDF。（Wind Tool 慣例）

## 12. 驗證習慣（交付前自檢）

1. **每個 UI 改動用 headless browser 實際驗證**再交付，commit message 記錄驗證
   情境與結果；互動式拖曳等無法 headless 的項目明確標註「需實機確認」。
   （Report Builder 後期 commit 全部照此格式）
2. **改 A 功能時檢查共用元件的其他呼叫端**（如共用選取器改通用後，確認比對頁
   無回歸）。
3. **squash-merge 後檢查有沒有殘留舊 UI**（重複按鈕曾因此漏進 main）。
   （5G-RRU PR #72）
